# MDO Project

## Структура проекта

- **MDO.Core**  
  Основная логика работы с базой данных:  
  - Управление подключением к базе данных  
  - Выполнение SQL-запросов  
  - Методы для открытия подключения, получения версии MS SQL Server и закрытия подключения

- **MDO.Api**  
  Веб-API, предоставляющий доступ к функционалу Core через HTTP:  
  - `ConnectToDatabase()`  
  - `GetSqlVersion()`  
  - `CloseDatabaseConnection()`  

  Используется **ASP.NET Web API**, удобно для интеграции с клиентскими приложениями.

- **MDO.Desktop**  
  Десктопное приложение, взаимодействующее с Web API:  
  - Подключение к backend API  
  - Вызов методов для получения версии SQL Server  
  - Отображение статуса подключения  
  - Закрытие подключения через UI

**Решение разделено на три проекта**, чтобы чётко отделить бизнес-логику, сервисный слой и клиентский интерфейс. Такая структура улучшает поддерживаемость и тестируемость.

## Используемые технологии

- **C# .NET Framework** (Core и Desktop)  
- **ASP.NET Web API**  
- **MS SQL Server**  
- **WinForms**

## Как запустить

1. Откройте решение в **Visual Studio**  
2. Постройте решение для восстановления пакетов NuGet  
3. Запустите проект `MDO.Api` для хостинга веб-сервиса  
4. Запустите проект `MDO.Desktop` для запуска клиентского приложения  
5. Используйте десктопное приложение для подключения к сервису и работы с базой данных

## Технические решения

- Проект использует **ASP.NET Web API** для сервисного слоя.  
  В реальных проектах я обычно предпочитаю **ASP.NET Core Web API** для лучшей производительности, кроссплатформенности и современных возможностей.

- Для работы с базой данных в профессиональных проектах я чаще использую **Entity Framework (EF) Core** или **Dapper** для более простой и безопасной работы с данными.  
  В этом задании я **не использовал EF или Dapper**, так как задача требовала напрямую управлять подключением к SQL Server, что обеспечивается самим фреймворком.  
  Использование EF абстрагировало бы эти детали и уменьшило видимость жизненного цикла подключения, что является ключевой частью задания.

- Основная цель — продемонстрировать **чёткое управление подключением**, получение версии SQL Server и закрытие соединения, а не полноценный ORM-слой.

- В реальных проектах также рекомендуется добавлять **Unit Tests**, однако в этом задании они **не были включены**, так как не требовались.
